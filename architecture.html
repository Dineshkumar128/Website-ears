How EARS is
put together

A single-origin, fully client-side application with no build step and no server dependency — designed for longevity and clean handoff.

The big picture

Layered client architecture

Every layer lives in the browser. The server serves static files — all logic, storage and caching is client-side.

All state in localStorage
The DB object wraps localStorage with typed helpers. Users, activities, audit logs and sessions persist across sessions.

🔄
Migration-ready design
All data access routes through the DB helper object. Swapping to Supabase or a REST API means updating only that one object — the rest of the app is untouched.
🔑
Auto-incrementing IDs
DB.nextId(key) manages its own counter per collection — mimics a database SERIAL sequence with zero dependencies.
📋
Audit log (capped)
Every action is logged — capped at 200 entries automatically to prevent localStorage overflow.
Design decision

Why no framework?

With a framework
500KB+ runtime bundle · Build chain required · npm install before anything works · Dependency security surface · Framework version lock-in · Requires Node.js environment
Vanilla — what we chose
Loads in <50ms · Edit and save — no build step · Zero dependencies · 100/100 Lighthouse performance · Works on any web server · Handoff with no setup friction
⚡
Zero build chain
Any developer with a text editor can pick it up and run it immediately — the entire codebase is edit-and-save.
♾️
Expandable by design
The DB layer, sync module, and auth functions are designed as seams — any piece can be swapped independently.
🌐
Deploys everywhere
Static files deploy to Netlify, GitHub Pages, S3 or any web server. No runtime, no database. Infrastructure cost: $0.
