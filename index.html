Background

A real problem. A practical solution.

University staff are required to record engagement activities — industry meetings, guest lectures, community events, advisory work — for performance reviews and institutional reporting. There was no system to do this reliably.

Before EARS

Staff relied on personal spreadsheets, informal notes or memory. There was no approval workflow, no audit trail, and no way to enforce who could change what. Any record could be edited or deleted without trace — a serious limitation for compliance and performance reporting.

After EARS

A structured submission and approval process, a tamper-evident audit log, and offline access on any device — all delivered as a Progressive Web App with a single codebase. Simple enough for any staff member without training, structured enough for institutional compliance reporting.

Why PWA, not a native app?
A native app requires separate iOS and Android codebases plus App Store approval and fees. A PWA distributes via a URL, works on every device instantly, costs nothing to publish — ideal for a university staff tool needing fast, free reach.
Why no frameworks?
Pure HTML, CSS and JavaScript — no dependencies. A future developer can read and modify the codebase without learning a specific framework. It will work in any modern browser regardless of what happens to the JavaScript ecosystem around it.
Why Netlify hosting?
Setting up and managing a dedicated server would consume time better spent on interface and documentation. Netlify provides free hosting, automatic HTTPS, Git-based deployment and native PWA support — no server configuration required.

What was built — Sprint 4

Every requirement delivered.

All must-have requirements from the MoSCoW analysis met and approved by the client. Built with zero external libraries.

01
Login & registration
Staff register and sign in securely with two roles — Staff and Administrator. Password rules enforced. Demo accounts available for evaluators.
02
Submit & manage activities
Staff log title, type, date, organisation, location, hours, description and whether in-person or online. They can edit or delete their own records.
03
Administrator approval
Administrators view all submitted activities and approve or reject them — a clear, accountable workflow with full visibility across all staff.
04
Tamper-evident audit log
Every action logged with timestamp and user identity. The log cannot be edited or deleted — a genuine compliance record of all system activity.
05
Dashboard & timeline
Activity summary by type, monthly counts, organisations engaged and upcoming events. Chronological timeline view, sortable and filterable.
06
Reports, export & offline PWA
Filter and print activity reports. Export all data as a portable JSON file for backup or sharing. Installs on any device, works without internet.

AI-assisted development

We used AI. Here is exactly how.

Two AI tools were used throughout the project. The critical distinction: AI was an implementation tool, not a decision-maker. All requirements, design decisions, testing and product judgements were the team's own. Nothing entered the codebase without explicit review and approval.

Claude (Anthropic)
claude.ai — conversational assistant
Used for architecture reasoning, feature design, complex logic implementation, debugging and documentation drafting. Suited for tasks requiring explanation and reasoning, not just pattern completion.
"The student typed a prompt describing a requirement. Claude proposed code. The student reviewed it, tested it in the browser, and either accepted it, required modifications, or rejected it entirely."
GitHub Copilot
VS Code inline extension
Inline code completion and boilerplate generation as the student typed. Used for repetitive patterns, standard functions and syntax assistance. Each suggestion was accepted or dismissed individually — one line at a time.
"All AI suggestions were reviewed, never automatically applied. Every change was tested in the browser and accepted only after the team confirmed correct behaviour."

Case study — the highest-level decision in the project
Google Sheets sync: built, tested, and removed by the team
The team built an auto-sync feature to push activity data to a Google Sheet. Testing revealed persistent CORS failures when the app was served from a local file context. Claude continued proposing technical workarounds — JSONP callbacks, no-cors fetch mode. The team tested each one. None were reliable. The decision to remove the feature entirely was the team's own product judgement — the AI never suggested it. This is the clearest illustration of the distinction between AI-assisted implementation and human product thinking.

Where human oversight was non-negotiable
!

!
Security gaps — plain-text password storage not flagged by AI. Caught by the team through deliberate code review.
!
